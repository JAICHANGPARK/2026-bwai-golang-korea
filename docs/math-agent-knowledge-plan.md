# 중고등학생 대상 수학문제 해설 에이전트 기획 및 지식 데이터 수집 정리

작성일: 2026-04-04  
작성 기준 시점: 한국 교육과정 및 수능 운영 현황 기준

## 1. 문서 목적

이 문서는 중고등학생 대상 수학문제 해설 에이전트를 만들기 전에 필요한 지식 데이터 수집 방향, 교육과정 개정 비교, 수능 데이터 활용 전략, 저작권 및 운영 주의점, 그리고 에이전트 설계 방향을 한 번에 정리한 문서다.

### 1.1 문서 적용 범위

이 문서는 `제품 준비 및 확장 단계`를 위한 기획 문서다.  
지식 데이터 수집, RAG, File Search, 교육과정 매핑 같은 중장기 제품 요소를 포함한다.

따라서 1시간 핸즈온 범위를 정의하는 기준 문서로 쓰면 안 된다.  
핸즈온에서는 `hands-on-math-agent-session-guide.md`와 `docs/prompt-pack/`를 기준으로 삼고, 이 문서는 세션 이후 확장 로드맵 참고 자료로만 사용한다.

목표는 다음과 같다.

- 사용자가 사진으로 업로드한 수학문제를 읽는다.
- 문제를 해석하고 풀이한다.
- 정답을 도출한다.
- 풀이 과정을 단계별로 설명한다.
- 이 문제가 무엇을 보려는 문항인지 설명한다.
- 어떤 교육과정과 성취기준, 어떤 단원과 연결되는지 안내한다.
- 부족한 선행 개념과 복습할 단원을 추천한다.

## 2. 에이전트의 최종 목표

이 에이전트는 단순한 `정답 생성기`가 아니라 아래 기능을 함께 제공하는 `문항 해설형 학습 에이전트`를 목표로 해야 한다.

- 문제 이미지 이해
- 수식 및 도형 인식
- 문항 구조화
- 풀이 생성
- 계산 검산
- 출제 의도 설명 또는 의도 추정
- 교육과정/성취기준 매핑
- 오답 포인트 분석
- 복습 단원 추천

추천 출력 형식은 아래와 같다.

1. 문제 해석
2. 정답
3. 풀이 과정
4. 핵심 개념
5. 이 문제의 의도
6. 자주 하는 실수
7. 연결 교육과정/성취기준
8. 복습 추천

중요한 점은 `출제자의 실제 의도`와 `문항이 평가하려는 핵심`을 구분하는 것이다. 공식 문서에 명시된 경우가 아니면 서비스 문구는 `출제 의도 추정`, `이 문항이 보려는 핵심`, `평가 포인트`처럼 표현하는 것이 안전하다.

## 3. 왜 지식 데이터를 먼저 모아야 하는가

수학 해설 에이전트는 모델 하나만 붙인다고 바로 품질이 나오지 않는다. 특히 교육용 서비스는 다음 요소가 같이 있어야 품질이 안정된다.

- 정확한 문제 텍스트와 수식
- 믿을 수 있는 정답
- 단계별 풀이 데이터
- 교육과정 기준점
- 개념 태그와 선수학습 관계
- 오답 패턴
- 검산 가능한 수학 엔진 또는 규칙

즉, 핵심 자산은 `문항 + 정답 + 해설 + 교육과정 + 개념 관계망`이다.

### 4.2 강력 추천 데이터

- 문항별 개념 태그
- 자주 틀리는 오답 유형
- 단계별 풀이 분해 데이터
- 난이도
- 출제 포인트 또는 평가 요소

### 4.3 나중에 직접 구축해야 할 데이터

공식 자료만으로는 아래 항목이 충분하지 않다.

- 출제 의도 요약
- 학생 눈높이별 풀이 버전
- 오답 원인 분류
- 선행 개념 부족 진단
- 맞춤형 복습 추천

이 부분은 결국 내부 라벨링 체계가 필요하다.

## 5. 추천 데이터 소스

### 5.1 교육과정/성취기준

가장 중요한 기준점은 교육부 고시와 NCIC다.

- 교육부 `2022 개정 교육과정` 고시
- 교육부 `2015 개정 교육과정` 관련 자료
- NCIC 성취기준/교육과정 자료

이 데이터는 에이전트가 문제를 단원, 과목, 성취기준에 연결할 때 기준이 된다.

### 5.2 수능 및 모의평가

고등 수학용 고난도 추론 데이터로 매우 중요하다.

- KICE 대학수학능력시험 문제 및 정답
- 6월/9월 모의평가 문제 및 정답
- 이의신청 심사 결과, 출제 방향 관련 자료

수능 데이터는 반드시 포함하는 것이 좋다. 다만 이것만으로는 부족하다. 이유는 아래와 같다.

- 중학교 데이터가 비어 있다.
- 내신형/학교형 서술 문제가 부족하다.
- 저중난도 학습 설명 데이터가 부족하다.
- 사진 촬영형 데이터와는 결이 다르다.

### 5.3 AI-Hub

학습용 데이터셋으로 유용하다.

- 수학 교과 문제 풀이과정 데이터
- 수학 과목 문제 생성 데이터

이 데이터는 문제 이미지, 정답, 풀이, 성취기준 매핑, OCR/수식 구조화 등에 도움된다.

### 5.4 보강 데이터

- 교과서/학습서 기반 개념 설명
- 자체 제작 오답노트형 데이터
- 교사 검수 풀이 데이터
- 학생 실제 업로드 이미지 샘플

## 6. 수능 데이터는 왜 꼭 수집하는가

수능과 KICE 모의평가 데이터는 아래 이유로 가치가 크다.

- 문항 품질이 높다.
- 추론형 문항 구조가 정교하다.
- 오답 선택지 설계가 좋다.
- 난이도 분포와 변별 포인트를 보기 좋다.
- 풀이 과정 설명 모델의 고난도 성능을 끌어올리는 데 유리하다.

다만 운영상 주의가 필요하다.

- 수능은 `고등 심화 문제`에 편향된다.
- 중학교/기초 설명형 데이터로는 부족하다.
- 원문 이미지 자체를 서비스에 재배포하는 방식은 저작권 검토가 필요하다.

추천 활용 방식은 다음과 같다.

- 내부 학습/평가용 데이터로 사용
- 문항 구조와 해설 패턴 학습에 활용
- 공개 서비스에서는 원문 재배포보다 `문항 정보 구조화 결과`와 `자체 해설` 중심으로 제공

## 7. 교육과정 개정 비교

### 7.1 큰 흐름 요약

### 2009 개정

- 운영 유연화의 성격이 강하다.
- 학년군, 교과군, 집중이수제, 블록타임제 같은 운영 변화가 핵심이다.
- 과목 체계가 비교적 세분되어 있다.

### 2015 개정

- 문·이과 통합이 핵심이다.
- 핵심 개념 중심으로 내용을 정선했다.
- 학생 활동 중심 수업, 핵심역량, 공통 과목 강화가 중요하다.
- 현재 수능 체계의 기반이다.

### 2022 개정

- 고교학점제와 학생 선택권 확대에 더 맞춰져 있다.
- 언어·수리·디지털 기초소양 강화가 강조된다.
- 탐구와 개념 기반의 깊이 있는 학습을 지향한다.
- AI·디지털 시대에 맞춘 수학 과목 재구조화가 진행된다.

---

### 7.2 서비스 관점에서의 차이

| 구분 | 2009 개정 | 2015 개정 | 2022 개정 |
| --- | --- | --- | --- |
| 핵심 키워드 | 운영 유연화, 집중이수 | 문·이과 통합, 핵심개념 | 학점제 기반 선택 확대, 디지털 소양 |
| 서비스 영향 | 오래된 기출의 과목명 매핑 필요 | 현행 수능 기출 해석의 기준 | 최신 학생 학습 경로 추천의 기준 |
| 데이터 설계 포인트 | 과목명/범위가 현재와 다름 | 수능/모평 태깅 기본축 | 현재 학교 현장 복습 추천 축 |
| 주의점 | 과목명이 지금과 바로 연결되지 않음 | 고3, 수능 데이터와 직접 연결됨 | 중1·중2·고1·고2 현행 학습 기준 |

### 7.3 적용 시기 정리

`2026-04-04` 기준으로는 교육과정이 혼재되어 있다.

- 2025-03-01: 2022 개정 교육과정이 중1, 고1에 적용
- 2026-03-01: 2022 개정 교육과정이 중2, 고2에 적용
- 2027-03-01: 2022 개정 교육과정이 중3, 고3에 적용

즉 현재는 아래와 같이 보는 것이 맞다.

- 중1·중2·고1·고2: 2022 개정
- 중3·고3: 2015 개정

서비스는 당분간 반드시 `2015 개정 + 2022 개정 동시 지원` 구조여야 한다.

### 7.4 고등 수학 과목 체계 비교

아래 표는 서비스 설계에 직접 영향을 주는 `대표 과목 체계` 기준 정리다. 일부 전문/심화 과목은 학교별 운영 차이가 있어 대표 중심으로 묶었다.

| 개정 | 대표 고등 수학 과목 체계 | 비고 |
| --- | --- | --- |
| 2009 개정 | 기초 수학, 수학 I, 수학 II, 확률과 통계, 미적분 I, 미적분 II, 기하와 벡터, 고급 수학 I, 고급 수학 II | 오래된 기출/평가자료 매핑 시 중요 |
| 2015 개정 | 수학(공통 과목), 수학 I, 수학 II, 미적분, 확률과 통계, 기하, 실용 수학, 경제 수학, 수학과제 탐구, 심화 수학 I, 심화 수학 II | 현재 고3 및 현행 수능 체계 이해의 기준 |
| 2022 개정 | 기본수학1, 기본수학2, 공통수학1, 공통수학2, 대수, 미적분Ⅰ, 확률과 통계, 미적분Ⅱ, 기하, 경제 수학, 인공지능 수학, 직무 수학, 수학과 문화, 실용 통계 | 최신 학교 현장과 2028 이후 수능 구조 이해에 중요 |

2022 개정에서는 추가로 과학 계열 선택 과목으로 `이산 수학`, `전문 수학`, `고급 대수`, `고급 미적분`, `고급 기하`가 확인된다.

### 실무 해석

- `2009 개정` 데이터는 과목명부터 현재 체계와 다르다.
- `2015 개정`은 현재 수능/모평 기출 해석의 핵심 축이다.
- `2022 개정`은 학생이 지금 배우는 과목 체계와 학습 추천의 핵심 축이다.

예를 들어 예전 기출을 설명할 때는 아래처럼 두 축을 같이 보여주는 것이 좋다.

- 원 출제 기준: 2015 개정
- 현재 복습 추천 기준: 2022 개정

---

### 7.5 2015와 2022의 차이를 에이전트에서 어떻게 처리할까

문항 하나에 아래 정보를 따로 저장하는 것이 좋다.

- `source_curriculum_version`: 문항이 실제 출제된 교육과정 버전
- `current_learning_curriculum_version`: 현재 학생에게 연결할 복습 기준 버전

예시:

- 2025년 11월 13일 시행 `2026학년도 수능` 문항
- 실제 출제 기준: `2015 개정`
- 현재 고1/고2 학생에게 추천할 복습: `2022 개정` 기준으로 연결

## 8. 현재 시점 기준 수능과 교육과정

### 8.1 수능 학년도와 시행일

헷갈리기 쉬운 부분이라 절대 날짜로 정리한다.

- `2026학년도 수능` 시행일: `2025-11-13`
- 교육부 발표 기준: `2015 개정 교육과정`에 따라 출제

즉 학년도와 실제 시험 날짜는 다르다. 데이터 스키마에는 반드시 두 필드를 분리해야 한다.

- `exam_year_label`: 2026학년도
- `exam_date`: 2025-11-13

### 8.2 2028 이후 수능과 연결되는 포인트

교육부 2028 대입 개편 Q&A 기준으로, 현행 수능은 대체로 아래 구조다.

- 공통: 수학 I, 수학 II
- 선택: 확률과 통계, 미적분, 기하 중 택1

2028 이후에는 아래 방향이 제시되어 있다.

- 대수
- 미적분Ⅰ
- 확률과 통계

이 변화는 향후 데이터 수집과 평가셋 설계에도 직접 영향을 준다.

## 19. RAG / GraphRAG 검토

사용자 질문 흐름을 기준으로 보면, 이 서비스에서 RAG와 GraphRAG는 `문제 풀이 엔진`이라기보다 `설명과 학습 연결 엔진`에 가깝다.

핵심 판단은 다음과 같다.

- `RAG는 도입하는 것이 좋다.`
- `GraphRAG는 나중에 선택적으로 붙이는 것이 좋다.`
- `문제 풀이 자체는 RAG보다 이미지 이해, 문제 구조화, 수학 검산이 더 중요하다.`

### 19.1 왜 RAG가 필요한가

RAG는 아래 상황에서 특히 유용하다.

- 어떤 성취기준과 연결되는지 설명할 때
- 어떤 단원과 개념을 복습해야 하는지 추천할 때
- 같은 유형의 대표 풀이 전략을 찾아 설명할 때
- 오답 포인트를 붙일 때
- 해설 문체와 설명 기준을 일관되게 유지할 때

즉, RAG는 `정답을 대신 찾는 도구`보다는 `해설을 더 교육적으로 만드는 도구`다.

### 19.2 왜 GraphRAG는 초기에 과한가

GraphRAG는 관계 탐색이 필요한 경우에 강하다.

- 선수학습 추적
- 개념 간 관계 탐색
- `2015 개정 문제 -> 2022 개정 복습 단원` 연결
- 학생별 약점 경로 분석
- 오답 원인에서 선행 부족 개념까지 역추적

하지만 초기에는 아래 이유로 비용이 크다.

- 그래프 스키마를 먼저 잘 설계해야 한다.
- 관계 라벨링 품질이 낮으면 오히려 품질이 흔들린다.
- 이미지 인식과 풀이 정확도보다 체감 효과가 늦게 난다.

그래서 1차 버전은 `벡터 RAG + 메타데이터 필터`가 더 현실적이다.

### 19.3 추천 원칙

이 서비스는 `retrieve first`보다 `solve first, retrieve second`가 더 맞다.

권장 흐름:

1. 사용자가 문제 이미지 업로드
2. 문제를 읽고 구조화
3. 문제를 풀이하고 검산
4. 그 다음에 관련 개념/성취기준/복습 내용을 검색
5. 최종 해설 생성

이 순서가 중요한 이유는, 문제를 풀기 전에 유사문항 검색에 너무 의존하면 틀린 문항이나 다른 유형에 끌려갈 위험이 있기 때문이다.

### 19.4 지금 시점의 추천

- 초기 MVP: `RAG 도입`
- 초기 MVP: `GraphRAG는 보류`
- 2차 버전: `개념 그래프`, `학습 경로 추천`, `개인화 약점 분석`이 필요할 때 GraphRAG 추가

## 20. RAG를 쓰려면 왜 사전 지식 데이터가 필요한가

RAG는 검색 인프라만 만든다고 바로 품질이 나오지 않는다. 검색할 수 있는 지식 코퍼스가 먼저 있어야 한다.

중요한 점은 `수학 데이터를 전부 모으는 것`이 아니라, `실제로 검색에 쓸 수 있는 형태로 정규화된 데이터`를 만드는 것이다.

### 20.1 필요한 데이터 층위

- 원천 데이터 저장소
  - PDF
  - 문제지
  - 정답
  - 해설
  - 교육과정 원문
- 정규화 DB
  - 문항
  - 개념
  - 성취기준
  - 교육과정 버전
  - 과목
  - 단원
- RAG 코퍼스
  - 개념 카드
  - 성취기준 카드
  - 오답 포인트 카드
  - 유형별 풀이 카드
  - 복습 추천 카드
- 그래프 DB
  - 선수학습 관계
  - 개념 관계
  - 교육과정 버전 간 매핑

### 20.2 RAG용 데이터는 어떻게 만들어야 하나

PDF 원문을 통째로 넣는 것은 좋지 않다. 아래처럼 짧고 의미 있는 검색 단위가 필요하다.

- 성취기준 1개
- 개념 설명 1개
- 오답 패턴 1개
- 대표 풀이 전략 1개
- 복습 추천 1개

권장 예시는 다음과 같다.

```json
{
  "id": "std_2022_high_commonmath1_set",
  "type": "achievement_standard",
  "text": "집합의 뜻과 표현을 이해하고 포함 관계를 판단할 수 있다.",
  "metadata": {
    "curriculum": "2022",
    "school_level": "high",
    "subject": "공통수학1",
    "concepts": ["집합", "포함관계"]
  }
}
```

### 20.3 지금 우선 구축할 것

- 2015/2022 교육과정 원문
- 성취기준
- 개념 설명 카드
- KICE 수능/모평 문제-정답-해설 메타데이터
- 문항별 개념 태그
- 오답 유형
- 복습 추천 매핑

즉, `RAG를 할지 말지`보다 먼저 `RAG에 넣을 지식 데이터를 어떻게 수집하고 정규화할지`를 설계해야 한다.

- 모델 capability와 lifecycle을 정기 점검할 것

즉, 지금 선택 자체는 괜찮지만 `preview`라는 사실은 반드시 운영 리스크로 관리해야 한다.

---

## 22. Gemini를 이 서비스에 어떻게 써야 하나

핵심은 `Gemini가 문제를 읽고 푸는 역할`과 `검색/RAG가 해설을 보강하는 역할`을 분리하는 것이다.

### 22.1 Gemini가 맡을 역할

- 문제 이미지 읽기
- 문제 구조화
- 풀이 초안 작성
- 풀이 과정 설명
- 사용자 눈높이에 맞는 자연어 해설 생성

### 22.2 Gemini에 붙일 기능

- `Code execution`
  - 계산 검산
  - 식 변형 검산
  - 수치 대입 검증
- `Structured outputs`
  - 정답
  - 풀이 단계
  - 개념 태그
  - 오류 가능성
- `File search` 또는 외부 RAG
  - 성취기준 검색
  - 개념 설명 검색
  - 복습 추천 검색

### 22.3 핵심 설계 원칙

- Gemini가 `문제를 푸는 메인 엔진`
- RAG는 `해설과 교육과정 연결을 보강하는 보조 엔진`
- 검산 계층은 반드시 별도로 둔다

즉, `문제 풀이를 RAG에 맡기면 안 되고`, `Gemini + 검산`이 문제 풀이의 중심이 되어야 한다.

## 23. ADK-Python / ADK-JS 활용 검토

사용 예정 오케스트레이션은 `adk-python`, `adk-js`다.

공식 ADK 문서 기준으로 ADK는 다음에 적합하다.

- 다중 에이전트 구성
- 워크플로 기반 실행
- 순차/병렬 처리
- 모델 및 툴 연결
- 구조화된 에이전트 제어

### 23.1 왜 ADK가 잘 맞는가

이 프로젝트는 완전 자유형 채팅보다 아래 특성이 중요하다.

- 흐름이 일정해야 한다.
- 단계별 실패 지점을 분리해야 한다.
- 검산/검색/설명 생성 역할을 분리해야 한다.
- 예측 가능한 응답 포맷이 중요하다.

이 점에서 ADK의 workflow agent 구조가 잘 맞는다.

### 23.2 Python과 JS를 어떻게 나눌까

권장 방향:

- `adk-python`
  - 메인 백엔드
  - 데이터 수집/정제
  - OCR 후처리
  - 수학 검산
  - 지식 데이터 구축
  - 메인 오케스트레이션
- `adk-js`
  - 프론트엔드 근접 로직
  - 웹앱 연동
  - UI 이벤트/에이전트 연결
  - 필요 시 경량 오케스트레이션

실무적으로는 Python을 메인으로 두는 편이 더 안정적이다. 이유는 수학 계산, 데이터 파이프라인, OCR/문서 처리, 후처리 생태계가 Python 쪽이 훨씬 강하기 때문이다.

## 24. 추천 에이전트 구성

ADK를 쓸 경우 아래처럼 역할 분리를 권장한다.

### 24.1 Orchestrator Agent

- 전체 요청 흐름 제어
- 어떤 하위 에이전트를 호출할지 결정
- 최종 응답 조합

### 24.2 Vision / Parse Agent

- 업로드 이미지 분석
- OCR 결과 정리
- 수식/도형/조건 추출
- 문제 구조화

### 24.3 Solver Agent

- 문제 풀이 초안 생성
- 풀이 전략 선택
- 정답 도출

### 24.4 Verifier Agent

- Code execution 또는 외부 계산기로 검산
- 풀이 오류 탐지
- 최종 정답 검증

### 24.5 Curriculum Agent

- 교육과정 버전 판별
- 성취기준 매핑
- 관련 단원/개념 검색

### 24.6 Explainer Agent

- 학생 수준에 맞는 해설 생성
- 출제 의도 또는 평가 포인트 설명
- 자주 하는 실수 안내
- 복습 추천 생성

### 24.7 Retriever Agent

- File Search 또는 외부 RAG 호출
- 개념 카드, 성취기준 카드, 오답 카드 검색
- 필요한 근거만 반환

## 25. 추천 처리 흐름

가장 현실적인 파이프라인은 아래와 같다.

1. 사용자가 문제 사진 업로드
2. Vision / Parse Agent가 문제를 구조화
3. Solver Agent가 풀이 초안 생성
4. Verifier Agent가 검산
5. Retriever / Curriculum Agent가 관련 지식 검색
6. Explainer Agent가 최종 해설 생성
7. Orchestrator Agent가 응답 포맷 정리

즉, `이미지 해석 -> 풀이 -> 검산 -> 지식 검색 -> 해설 생성` 흐름이다.

이 순서가 좋은 이유:

- 풀이 정확도를 먼저 확보할 수 있다.
- 검색이 정답 생성에 과도하게 개입하지 않는다.
- 교육과정 연결을 안정적으로 붙일 수 있다.

## 27. 최종 결론

지금까지의 논의를 종합하면 다음과 같다.

- 이 프로젝트는 충분히 AI 에이전트 구조로 구현 가능하다.
- `Gemini 3 Flash Preview`는 멀티모달 입력과 툴 사용성 때문에 매우 잘 맞는 선택이다.
- 하지만 문제 풀이 품질의 중심은 `이미지 이해 + 문제 구조화 + 검산`이다.
- RAG는 반드시 도움되지만, 역할은 `정답 생성`보다 `해설 강화`에 가깝다.
- GraphRAG는 당장 필수는 아니고, 2차/3차 버전에서 도입하는 것이 좋다.
- ADK는 역할 분리형 에이전트 구성에 잘 맞고, 메인 백엔드는 `adk-python`이 더 유리하다.

따라서 현재 가장 좋은 방향은 아래와 같다.

1. 지식 데이터 수집 및 정규화
2. Gemini 기반 문제 풀이/검산 파이프라인 구축
3. RAG 기반 교육과정/개념/복습 연결
4. 이후 GraphRAG와 개인화 학습 추천 확장
