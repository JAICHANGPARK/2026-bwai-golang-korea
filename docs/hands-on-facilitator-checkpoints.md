# 핸즈온 진행자 체크 포인트

이 문서는 `prompt-pack`과 분리된 진행자용 운영 메모다.  
참석자에게는 기본적으로 공유하지 않고, 세션을 이끌 때만 참고한다.

## 01. 사용자 흐름

- `풀이`와 `검증`이 분리되어 있는가
- RAG 없이도 흐름이 닫혀 있는가
- 너무 많은 기능이 들어가지 않았는가

## 02. 에이전트 역할

- Solver가 최종 승인자처럼 설계되지 않았는가
- Verifier가 설명자가 아니라 검토자로 설계되었는가

## 03. 시스템 프롬프트

- 출력 형식 강제가 들어갔는가
- 이미지가 불분명할 때 추측 금지 규칙이 들어갔는가
- Verifier가 충분히 엄격한가

## 03a. 화면 설계

- Solver / Verifier / Final 카드가 분리되어 있는가
- 업로드와 실행 버튼이 명확한가
- 너무 복잡한 대시보드형 UI가 아닌가

## 04. 출력 스키마

- UI에서 바로 쓰기 쉬운 구조인가
- `approved`, `revised`, `needs_clarification` 상태값이 있는가
- `steps`가 배열로 정의되어 있는가
- `approvedSolution`과 `nextAction`이 포함되어 있는가

## 05. 실패 처리

- 추측형 응답을 막는 규칙이 있는가
- 재업로드 요청 흐름이 있는가
- 실패 시에도 UX가 깨지지 않는가

## 05a. 최종 설명 계약

- raw Solver/Verifier JSON을 최종 설명으로 바로 노출하지 않는가
- 최종 설명 카드는 `approved` 이후에만 보이는가
- `revised`가 최종 답 표시가 아니라 재검토 상태로 처리되는가

## 06. 통합 명세

- Build와 Antigravity 양쪽에 모두 넣을 수 있을 만큼 일반적인가
- 기능 요구사항이 과도하게 많지 않은가
- 제약사항이 명확한가

## 07. Build 및 Antigravity

- 같은 명세를 두 도구가 모두 이해하는가
- 결과 차이를 비교할 수 있는가
- 참석자가 재사용 가능성을 느꼈는가

## 07a. Streaming 및 Markdown

- 스트리밍은 최종 사용자 설명에 집중되어 있는가
- JSON 구조와 스트리밍 UX가 충돌하지 않도록 분리했는가
- Markdown 렌더링이 읽기 쉽게 보이는가
