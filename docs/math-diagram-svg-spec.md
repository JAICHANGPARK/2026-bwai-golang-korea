# 수학 해설용 SVG 도형 설명 기능 정리

작성일: 2026-04-04  
버전: 0.1  
상태: Draft

## 1. 문서 목적

이 문서는 수학 문제 해설 시스템에 `SVG 기반 도형/그래프 설명` 기능을 어떻게 넣을지 정리한 제품 명세 초안이다.

핵심 목표는 다음과 같다.

- 문제 속 그림을 학생이 이해하기 쉬운 `해설용 시각 자료`로 다시 보여준다.
- 텍스트 해설만으로 어려운 `기하`, `함수 그래프`, `넓이/부피`, `단면` 문제를 더 잘 설명한다.
- 문제 원본 그림을 그대로 복제하는 대신, 풀이 이해를 돕는 `설명용 SVG`를 제공한다.

---

## 2. 결론

이 기능은 `도입하는 것이 좋다`.

특히 아래 유형에서 효과가 크다.

- 기하 도형 해석
- 함수 그래프 읽기
- 넓이와 부피 문제
- 단면 문제
- 좌표기하 문제

학생 입장에서는 정답 자체보다 `그림을 어떻게 읽어서 식으로 바꾸는지`가 더 어렵다.  
SVG 해설은 바로 그 지점을 시각적으로 연결해줄 수 있다.

---

## 3. 왜 필요한가

텍스트 해설만으로는 아래 문제가 자주 생긴다.

- 어떤 선분이나 높이가 중요한지 바로 보이지 않는다.
- 함수 아래 넓이, 사이 넓이, 단면 같은 개념이 머릿속에서 잘 그려지지 않는다.
- 문제 그림의 원근 표현이나 생략된 요소 때문에 학생이 핵심 구조를 놓친다.
- 해설이 맞더라도 학생은 `왜 그런 식이 나오는지`를 납득하지 못한다.

SVG 해설은 다음을 보완한다.

- 중요한 부분만 강조한 단순 도식 제공
- 식과 도형의 대응 관계 시각화
- 풀이 단계별 강조점 변경
- 모바일에서도 읽기 쉬운 깨끗한 그림 제공

---

## 4. 기본 원칙

### 4.1 원본 그림 복제가 아니라 해설용 도식화

시스템은 시험지 그림을 그대로 복원하려고 하지 않는다.  
대신 풀이에 필요한 관계만 남긴 `해설용 SVG`를 만든다.

예:

- 원근이 들어간 입체도를 그대로 복제하지 않음
- 핵심 밑면, 높이, 단면, 축, 구간만 남김

### 4.2 모델이 바로 최종 SVG를 쓰게 하지 않기

권장 흐름은 아래와 같다.

1. 문제를 이해한다.
2. 시각 설명이 필요한지 판단한다.
3. `diagram_spec`을 JSON으로 만든다.
4. 렌더러가 그 JSON을 바탕으로 SVG를 생성한다.

즉, `LLM 직접 SVG 생성`보다 `LLM -> 구조화 스펙 -> 렌더러` 구조가 안정적이다.

### 4.3 항상 필요한 기능은 아님

모든 문제에 SVG가 필요한 것은 아니다.

우선 적용 대상:

- 도형이 본질인 문제
- 그래프를 읽어야 하는 문제
- 면적/부피/단면 해석이 필요한 문제

적용하지 않아도 되는 대상:

- 단순 계산형 문제
- 짧은 대수 연산 문제
- 시각 정보가 거의 없는 문제

### 4.4 최종 설명과 연결되어야 함

SVG는 독립 그림이 아니라 `해설 단계`와 연결되어야 한다.

좋은 예:

- "이 구간이 밑면입니다."
- "이 길이가 정삼각형 한 변입니다."
- "이 함수값이 단면의 한 변 길이입니다."

## 5. 추천 처리 파이프라인

1. 사용자 입력 수신
2. 문제 구조화
3. 시각 설명 필요 여부 판단
4. `diagram_type` 분류
5. `diagram_spec` 생성
6. SVG 렌더링
7. 텍스트 해설과 SVG를 함께 표시

한 줄로 요약하면 아래와 같다.

`문제 이해 -> 시각화 필요 판단 -> diagram_spec 생성 -> SVG 렌더링 -> 해설과 결합`

---

## 6. 추천 아키텍처

### 6.1 주요 컴포넌트

- `Problem Parser`
  - 문제 텍스트
  - 수식
  - 도형/그래프 존재 여부
  - 구간, 축, 조건 추출

- `Diagram Planner`
  - 어떤 그림이 필요한지 결정
  - 어떤 요소를 강조할지 결정
  - `diagram_spec` 생성

- `SVG Renderer`
  - 정해진 스키마를 받아 SVG 생성
  - 스타일, 색, 라벨, 강조 반영

- `Explanation Composer`
  - SVG와 텍스트 해설을 연결
  - Markdown 카드 안에서 단계별로 보여줌

### 6.2 권장 분리

- 모델은 `의미 해석`을 담당
- 렌더러는 `정확한 그림 생성`을 담당

이 분리가 있어야 품질 검증과 디버깅이 쉬워진다.

## 7. diagram_spec 권장 스키마

최소 스키마 예시는 아래와 같다.

```json
{
  "needs_visual_explanation": true,
  "diagram_type": "cross_section_volume",
  "purpose": "explain_solution",
  "title": "단면과 밑면 해설",
  "coordinate_system": {
    "show_axes": true,
    "x_label": "x",
    "y_label": "y"
  },
  "elements": [
    {
      "kind": "curve",
      "label": "y = sqrt(x + x ln x)",
      "expression": "sqrt(x + x * ln(x))",
      "domain": [1, 2]
    },
    {
      "kind": "region",
      "label": "밑면",
      "source": "area_under_curve",
      "x_range": [1, 2],
      "lower_bound": "y=0"
    },
    {
      "kind": "cross_section",
      "label": "정삼각형 단면",
      "axis": "x",
      "shape": "equilateral_triangle",
      "side_length_source": "f(x)"
    }
  ],
  "highlights": [
    {
      "target": "cross_section",
      "message": "한 단면의 한 변 길이는 y값과 같다."
    },
    {
      "target": "region",
      "message": "x=1부터 x=2까지의 밑면을 본다."
    }
  ],
  "step_links": [
    {
      "step_index": 1,
      "target": "region"
    },
    {
      "step_index": 2,
      "target": "cross_section"
    }
  ]
}
```

## 8. 예시: 단면 부피 문제에 어떻게 쓰나

예시 문제처럼 `x=1`부터 `x=2`까지의 영역을 밑면으로 하고, `x축에 수직인 단면`이 모두 `정삼각형`인 경우 SVG는 아래를 보여주면 좋다.

- 함수 `y = sqrt(x + x ln x)`
- `x=1`, `x=2` 경계선
- 밑면이 되는 구간
- 대표 단면 하나
- 그 단면의 한 변이 `f(x)`라는 표시
- 단면 넓이 식 `sqrt(3)/4 * (f(x))^2`

이렇게 하면 학생은 아래를 더 쉽게 연결한다.

- 함수값 -> 단면 한 변 길이
- 단면 한 변 -> 정삼각형 넓이
- 단면 넓이 -> 적분 -> 부피

## 9. 우선 적용할 문제 유형

### 9.1 1순위

- 함수 아래 넓이
- 두 곡선 사이 넓이
- 단면과 부피
- 좌표평면 위 도형 해석

### 9.2 2순위

- 중학교 평면기하
- 합동/닮음
- 원과 접선
- 삼각비 해석

### 9.3 3순위

- 입체기하
- 공간좌표
- 벡터 도식
- 통계 그래프 해석

## 10. 렌더링 원칙

### 10.1 단순하고 설명 중심으로

- 장식보다 설명 우선
- 중요한 선과 영역만 남김
- 지나치게 복잡한 3D 표현 지양

### 10.2 강조가 분명해야 함

- 핵심 구간 색상 강조
- 중요한 높이, 변, 각 라벨 표시
- 단계별 강조 대상 변경 가능

### 10.3 모바일 친화

- 작은 화면에서도 라벨이 읽혀야 함
- 선 두께와 글자 크기 최소 기준 필요
- 너무 넓은 가로형 그림 지양

### 10.4 접근성

- 각 SVG에 짧은 설명 텍스트 제공
- 색만으로 의미를 구분하지 않음

## 11. UI 통합 방식

권장 방식은 아래와 같다.

- 최종 해설 카드 안에 SVG를 포함
- SVG는 관련 풀이 단계 바로 위나 옆에 배치
- SVG 아래에 짧은 캡션 제공
- 필요하면 단계 전환에 따라 강조 요소 변경

예:

1. 문제 해석
2. SVG로 밑면과 단면 표시
3. "이 길이가 단면의 한 변입니다" 설명
4. 식 전개
5. 최종 정답

### 11.1 Streaming과의 관계

최종 사용자 설명이 Markdown으로 스트리밍된다면, SVG는 아래 둘 중 하나로 처리하는 것이 좋다.

- 먼저 구조화된 SVG를 렌더링하고, 텍스트 설명을 스트리밍
- 텍스트 스트리밍 종료 후 SVG를 함께 확정 표시

처음 구현은 두 번째가 더 단순하다.

## 12. 비추천 방식

- 모델이 아무 검증 없이 raw SVG 전체를 바로 생성
- 문제 원본 그림을 픽셀 단위로 복제하려는 시도
- 모든 문제에 SVG를 강제로 붙이는 방식
- 텍스트 해설과 무관한 장식성 도형 추가

## 13. 주요 리스크

### 13.1 의미 오해

문제 그림을 잘못 읽으면 잘못된 도형을 설명하게 된다.

대응:

- `diagram_present`와 별도로 `diagram_confidence` 관리
- 저신뢰 시 SVG 생성 보류

### 13.2 과도한 일반화

모든 도형 문제를 하나의 범용 렌더러로 처리하려 하면 품질이 흔들릴 수 있다.

대응:

- 문제 유형별 렌더러 분리
- 우선순위 높은 유형부터 제한적으로 구현

### 13.3 해설과 그림 불일치

텍스트는 맞는데 SVG가 틀리거나, 반대로 SVG는 맞는데 설명이 어긋날 수 있다.

대응:

- `step_links`로 해설 단계와 SVG 요소를 명시적으로 연결
- 검증 단계에서 SVG 스펙도 함께 점검

## 14. 권장 구현 순서

### 14.1 1단계

- 함수 그래프
- 넓이 문제

이 단계에서는 제한된 타입만 지원하는 것이 좋다.

### 14.2 2단계

- 평면기하
- 좌표기하
- 삼각비 도형

### 14.3 3단계

- 입체기하
- 공간벡터
- 더 정교한 단계별 강조

---

## 15. 핸즈온과 제품 기준의 차이

### 15.1 핸즈온

- 선택 기능으로만 다룬다.
- 실제 구현보다 `설계 아이디어` 수준으로 소개한다.
- 필요한 경우 샘플 SVG 몇 개를 고정 자산으로 보여준다.

### 15.2 제품

- 적극 도입을 추천한다.
- `diagram_spec -> SVG renderer` 구조로 구현한다.
- Verifier가 도형 설명 적절성도 같이 점검하게 한다.

## 16. 최종 권장안

한 줄로 정리하면 아래와 같다.

`SVG 도형 해설은 매우 유용하고, 특히 그림을 식으로 바꾸는 과정을 보여주는 데 강하다. 다만 원본 복제형이 아니라 설명용 도식화 SVG로 가야 하고, 모델이 직접 SVG를 쓰기보다 구조화 스펙을 먼저 만들게 하는 것이 좋다.`
